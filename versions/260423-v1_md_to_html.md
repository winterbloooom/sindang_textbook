# Version 1 — MD → HTML 변환 방식

- 작성일: 2026-04-23
- 상태: 운영 중

## 개요

교재 한 장(章)을 만드는 파이프라인의 초판(version 1)이다.
작성자는 `content.md`에 교재 내용을 쓰고, 정해진 규칙(`template/RULES_conversion.md`, `template/RULES_design.md`)에 따라 Claude Code가 `content.html`로 변환한다. 결과물은 A4 인쇄를 전제로 한 템플릿에 맞춰진다.

관련 파일:
- `template/RULES_conversion.md` — MD 프런트매터/마크다운 → HTML 클래스 매핑 규칙
- `template/RULES_design.md` — 시각 스타일(팔레트·타이포·컴포넌트) 규칙
- `template/examples/sample.{md,html}` — 변환 전/후 레퍼런스 쌍
- `textbook/{과목}/{편}/{장}/content.{md,html,pdf}` — 실제 장별 산출물

## 의도

- **작성 단계는 가볍게**: 교재 내용을 쓰는 일은 마크다운이 가장 편하다. 문단, 번호 매기기, 굵은 글씨 정도만 쓰면 된다.
- **규칙만 잘 구조화하면 변환은 쉬울 것**: 컴포넌트(박스·그림·페이지 번호 등)를 프런트매터와 마크다운의 정해진 꼴로 표기해두면, 규칙 기반으로 HTML에 일관되게 매핑할 수 있을 것이라 봤다.
- **한 번의 변환으로 완성**: 작성자는 MD만 신경 쓰고, HTML은 부산물처럼 얻어지는 그림.

## 단점 (실제로 써보니)

### 1) 규칙 기반만으로는 깔끔하게 한 번에 전사되지 않는다
구조화를 해두어도 최종 결과는 "규칙 매핑 + 에이전트의 센스"가 결합되어 나온다.
- 같은 MD를 변환해도 페이지 나눔, 박스 경계, 그림 위치 같은 판단 영역이 늘 남는다.
- QA 스킬(`qa-*`)로 잡아내긴 하지만, 결국 재변환/수정 루프가 필요하다.

### 2) 변환 한 번에 토큰을 너무 많이 쓴다
한 장(章) 전체를 한 번에 읽고 쓰는 방식이라, 변환 1회당 입력·출력 토큰이 크다.
- 규칙 문서(`RULES_conversion.md`, `RULES_design.md`)와 예시 쌍까지 매번 같이 들어간다.
- 장이 길어질수록 비용이 선형 이상으로 늘어난다.

### 3) 사소한 변경도 Claude Code 호출이 필요하다
HTML을 직접 손보면 MD와 어긋나므로, 오타 하나, 줄바꿈 하나를 고치려 해도 원칙적으로는 MD를 고치고 다시 변환해야 한다.
- 변경 범위에 비해 호출 비용이 과하다.
- "작성자 혼자서 빠르게 고친다"가 잘 안 된다.

## 요약

Version 1은 **작성은 가볍게, 변환은 자동으로**라는 전제로 만들었지만, 실제로는 **변환이 자동이 아니라 반자동**이고, **비용은 작지 않으며**, **사후 수정이 비싸다**. 다음 버전은 이 세 가지를 줄이는 방향으로 가야 한다.
